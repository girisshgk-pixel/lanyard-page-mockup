Page Layout — Sticky Product Panel + Scrollable Content

UX / CRO
Single-column reading flow → Amazon-style sticky product panel

The current page renders all product content in a single scrolling column. The product image, pricing, and order CTA scrolled out of view as users read content — creating a disconnect between purchase intent and the order action.

The proposed design fixes the product image, title, pricing, size selector, and order CTA in a sticky left panel (position: sticky; top: 65px) that remains visible at all times while the right column scrolls through features, pricing tables, calculator, testimonials, and FAQ.

This mirrors the layout pattern used by Amazon, Shopify, and high-converting B2B product pages — keeping the purchase path permanently accessible without forcing the user to scroll back up.

Expected improvements
  • Add-to-cart rate (CTA always visible)
  • Time on page (content is explorable)
  • Bounce rate reduction (product anchored)
  • Mobile: panel collapses to top, thumbs go horizontal
Dev note The sticky panel width is set to 520px fixed. On viewports under 900px it collapses to grid-template-columns: 1fr. Ensure the WordPress theme wrapper does not apply overflow: hidden on the parent — this breaks position: sticky.

Image Gallery — 1080×1080 Main Image + 6 Shot-Type Thumbnails

UX / SEO
Small static image → 1080×1080 large viewer with vertical thumb strip & zoom modal

The current page displays a single product image at roughly 400px, embedded inline with the text. There were no thumbnails showing alternative views, no zoom capability, and no way for the visitor to inspect the product closely.

The proposed design implements a vertical thumbnail strip on the left (6 thumbnails) feeding a large 1:1 main image on the right. Clicking the main image opens a full-screen zoom modal. Each thumbnail represents a different shot type of the same product — not a different product variant:

  • 1 Wide-shot on white background
  • 2 Close-up fabric / weave texture
  • 3 Male lifestyle mockup (man wearing)
  • 4 Female lifestyle mockup (woman wearing)
  • 5 Hook / buckle / clip hardware close-up
  • 6 Custom logo / silk screen print close-up
Expected improvements
  • Google Image Search impressions (1080px images preferred)
  • ImageGallery rich result eligibility
  • Purchase confidence (product detail visible)
  • Return rate reduction (accurate expectations)
Action required Currently all 6 thumbnails point to the same source image (the one available on the live site). Badge-A-Minit's photographer or design team needs to produce the 5 additional shot types (fabric close-up, male/female mockups, hook detail, logo close-up) at 1080×1080px and replace the src URLs in the thumbnail and ImageGallery schema accordingly.

Programmatic Alt Text & Title — Per Shot Type, Per Colour

SEO
Accessibility
Generic static alt → dynamic shot-specific alt + title, auto-updating on switch

The current image has a single generic alt text that did not describe the shot, the context, the brand, or include any conversion-oriented signal. Title was absent entirely.

The proposed implementation assigns unique, descriptive alt and title text to every thumbnail, the main image, and each zoom state. When a user clicks a thumbnail or colour swatch, the switchImg() JS function updates mainImg.alt and mainImg.title programmatically so Google always reads the correct descriptor for whichever image is in view.

Heading Hierarchy — Strict H1 → H2 → H3 Enforcement

SEO
Accessibility
Multiple H1s / flat heading structure → single H1 with correct H2/H3 cascade

The current product page uses multiple H2 tags for both section titles and card-level sub-items, with no clear hierarchy. The page also lacked a dedicated H1 — the product title was rendered as a styled <div>.

Google's NLP systems use heading hierarchy to understand content structure and extract featured snippet candidates. A flat or broken hierarchy confuses entity association and reduces eligibility for rich results.

The proposed structure follows a strict cascade:

H1 (×1) — Polyester Lanyard (Silk Screen Printing)
H2 (×11) — Section titles (Overview, Features, Pricing…)
H3 (×21) — Sub-items (feature cards, step titles, FAQ Qs…)
Expected improvements
  • Featured snippet eligibility for FAQ answers
  • NLP entity clarity for "polyester lanyard" topic
  • Screen reader navigation (landmark structure)
  • Core Web Vitals: no H1 confusion for LCP scoring

Bulk Price Calculator — Live Slider with 250+ Volume Discount Reveal

UX / CRO
Static pricing table → interactive live calculator with savings display

The current page shows pricing in a static HTML table. Users had to mentally calculate their total and find their applicable tier — a known friction point for B2B buyers ordering in bulk.

The proposed calculator lets buyers select lanyard width, print side (single/double), drag a quantity slider (250–2,000), and instantly see unit price, total cost, and savings vs. single-unit rate. When quantity exceeds 2100 units, a green savings banner appears automatically.

The calculator exists in two places: inline in the sticky left panel (for immediate purchase decisions) and as a full-width section in the scrollable content area (for deeper exploration).

Expected improvements
  • Average order value (volume incentive visible)
  • Dwell time (interactive engagement)
  • Conversion rate (removes pricing friction)
  • Quote request rate (for larger quantities)
Pricing accuracy All pricing data is hardcoded from the live product page at time of build. If Badge-A-Minit updates pricing in their CMS, this file must be updated to match. Recommend linking the calculator to a live pricing API endpoint if available.

Proposed lanyard_product_page.html — live calculator JS 
const PRICES = {
  '1':   { s: [2.44,1.95,1.50,1.31,1.13,0.86],
           d: [2.54,2.03,1.56,1.37,1.17,0.90] },
  '1.5': { s: [2.54,2.03,1.56,1.37,1.17,0.90],
           d: [2.63,2.11,1.62,1.42,1.22,0.93] },
  // ... 2cm and 2.5cm rows
};
const TIERS = [100,250,500,1000,2500,5000,10000];

function getPrice(qty, size, side) {
  const p = PRICES[size][side];
  for (let i = TIERS.length-1; i >= 0; i--) {
    if (qty >= TIERS[i]) return p[i];
  }
  return p[0];
}

function updateCalc() {
  const qty   = parseInt(range.value);
  const price = getPrice(qty, cSize, cSide);
  const total = Math.round(price * qty);
  const saved = Math.round((cRetail - price) * qty);

  // Live DOM updates
  calcUnit.textContent  = '$' + price.toFixed(2);
  calcTotal.textContent = '$' + total.toLocaleString();
  calcSaved.textContent = '$' + Math.max(0,saved)
                               .toLocaleString();

  // Reveal savings banner at 250+ units
  if (qty >= 250) {
    banner.classList.add('on');
    bannerAmt.textContent = '$' + saved.toLocaleString();
  } else {
    banner.classList.remove('on');
  }
}

Robots Meta & Dynamic Canonical URL

SEO
No robots directive + static canonical → noindex+nofollow with max-preview + dynamic canonical

This is a proposed change for the staging/experimental version of the file. The robots meta is intentionally set to noindex, nofollow so Google does not index this version while it is being tested. However, the max-snippet:-1, max-image-preview:large, and max-video-preview:-1 directives are included so Google can still render full previews if it does crawl the page — useful for staging QA against Search Console.

The proposed canonical would be set dynamically via JavaScript to strip all UTM parameters, fbclid, gclid, and ref parameters before writing the canonical link tag — preventing duplicate content signals from campaign traffic.

Production deployment note When implementing: change noindex, nofollow to index, follow before going live. The max-preview directives should remain. The dynamic canonical script should remain as-is.
Current current — no robots directive, static canonical 
<!-- No robots meta tag at all -->

<!-- Static canonical — won't strip UTM params
     from campaign traffic URLs -->
<link rel="canonical"
  href="https://badge-a-minit.com.au/
  products/lanyards"/>
<!-- Also: canonical points to category page,
     not the individual product page -->
Proposed lanyard_product_page.html — robots + dynamic canonical 
<!-- Robots: noindex for staging,
     max-preview for SERP preview quality -->
<meta name="robots"
  content="noindex, nofollow,
  max-snippet:-1,
  max-image-preview:large,
  max-video-preview:-1"/>

<!-- Dynamic canonical — strips tracking params -->
<link rel="canonical" id="canonicalTag"
  href="...product/polyester-lanyard-..."/>

<script>
(function() {
  var url = new URL(window.location.href);
  // Strip all campaign & tracking params
  ['utm_source','utm_medium','utm_campaign',
   'utm_term','utm_content','ref',
   'fbclid','gclid'].forEach(function(p) {
    url.searchParams.delete(p);
  });
  document.getElementById('canonicalTag')
    .setAttribute('href', url.href);
})();
</script>

URL Strategy for Colour & Option Variants — Default Option 1, Scalable to Option 3

SEO
URL Architecture
How colour, width, and print-side selections are handled in URLs without creating duplicate content

When a visitor selects a colour (e.g. Navy Blue) or width (e.g. 2cm) on the product page, a URL decision must be made: does the browser update the address bar? If so, how? This choice directly affects duplicate content risk, Google crawl budget, and long-term SEO scalability.

Two approaches are documented. Option 1 is proposed as the default. Option 3 is available as a future upgrade path if search demand for individual colours grows.

✅ Option 1 — Query Parameters + Canonical (Default / Proposed)
/product/polyester-lanyard-silk-screen-printing?colour=green&width=1cm&side=single
/product/polyester-lanyard-silk-screen-printing?colour=navy&width=2cm&side=double

The base URL stays the same. Variant selections update the URL via query parameters. The dynamic canonical script already implemented strips these parameters and points all variants back to the clean base URL — preventing duplicate content from being indexed.

The colour and option data is communicated to Google through the color, size, and material fields in the Product and ItemList structured data — no separate pages needed.

🔭 Option 3 — Separate Canonical URLs Per Colour (Future / If Search Demand Warrants)
/product/polyester-lanyard-silk-screen-printing/green
/product/polyester-lanyard-silk-screen-printing/navy-blue
/product/polyester-lanyard-silk-screen-printing/red

Each colour gets its own URL with its own canonical pointing to itself. Upgrade to this model only when Google Search Console or keyword research shows meaningful search volume for colour-specific queries (e.g. "green custom lanyards Australia").

Each colour page would require: a unique hero image, colour-specific Product schema with its own image, a unique first paragraph mentioning the colour's common use case, and its own <link rel="canonical"> pointing to itself (not the parent).

Decision criteria for upgrading to Option 3
  • GSC shows 100+ monthly impressions for colour-specific queries
  • Client can supply unique photography per colour
  • CMS supports colour-slug URL routing cleanly
  • Content team can write unique copy per colour (min. 80 words)
Canonical script — add colour/option params to strip list The dynamic canonical already strips utm_*, fbclid, gclid. Ensure colour, width, and side are also added to the strip list so WooCommerce query parameters never leak into Google's canonical index.

Image Gallery
Product images: Single product URL with JS colour-swap vs. unique images per colour page

Option 1 (Default — Proposed): The colour swatch selector swaps the main product image via JavaScript — no page reload, no new URL. The ImageGallery schema lists the 6 shot types. All images reference the same product URL in their contentUrl.

Option 3 (Future): When colour-specific pages exist (per Section 13), each page hosts unique photography for that colour — the ImageGallery schema on the green page references green lanyard images only. This dramatically improves Google Image Search ranking for colour-specific queries.

Photography requirement for Option 3 Each colour needs: wide shot, fabric close-up, male mockup, female mockup, hook close-up, logo close-up — all photographed in that specific colour. This is a significant production cost — recommend only for the top 3 best-selling colours first.
